# Design Decisions

이 프로젝트에서 내린 핵심 설계 결정과 그 이유를 기록한다.

---

## 1. Notion OAuth 인증을 사용하는 이유

### 결정
Internal Integration Token 대신 **Notion OAuth (공개 통합)** 를 사용한다.

### 이유

**Internal Integration Token의 한계**
- 특정 워크스페이스에 수동으로 연결해야 함
- 토큰이 만료되지 않지만, 재발급 시 환경변수를 직접 수정해야 함
- 다른 워크스페이스로 이동하거나 계정을 변경할 경우 재설정이 번거로움

**OAuth의 장점**
- 표준 인증 흐름으로 연결/해제가 명확
- 향후 다른 Notion 워크스페이스 연결이 필요한 경우 재사용 가능
- 권한 범위를 명시적으로 선언 (read-only 명시)
- 개인 프로젝트에서도 올바른 인증 구조를 경험하는 학습 효과

### 트레이드오프
- 초기 설정이 Internal Token보다 복잡 (OAuth 앱 등록, redirect URI 설정 필요)
- 단기적으로는 Internal Token이 더 빠르지만, 구조적으로 OAuth가 더 명확함

## 2. Import-only (단방향) 구조를 선택한 이유

### 결정
이 서비스는 Notion 데이터를 **읽기(Import)만** 하며, 서비스에서 Notion으로 데이터를 쓰지 않는다.

### 이유

**신뢰할 수 있는 단일 소스**
- Notion이 모든 콘텐츠의 원천(source of truth)이다.
- 서비스와 Notion 간 데이터가 일치하지 않는 충돌 상황이 발생하지 않는다.
- 콘텐츠 수정은 항상 Notion에서만 이루어지므로 이력 관리가 명확하다.

**단순성**
- 양방향 동기화 로직은 충돌 해결, 버전 관리 등 복잡성을 수반한다.
- Import-only는 단순한 fetch → 캐시 → 렌더링 흐름으로 유지보수가 쉽다.

**기존 워크플로 유지**
- Notion의 편집기, 블록, 데이터베이스 기능을 그대로 활용할 수 있다.
- 서비스는 Notion 위에 "읽기 레이어"를 추가하는 역할만 한다.

## 3. 실시간 동기화를 하지 않는 이유

### 결정
Webhook, 폴링, 실시간 동기화를 사용하지 않는다. 사용자가 수동으로 Import를 실행한다.

### 이유

**복잡성 대비 효용이 낮음**
- 1인 개인 사용자 기준으로 콘텐츠 발행 빈도는 낮다.
- 실시간 동기화를 위한 Webhook 서버 운영 또는 폴링 cron job은 오버엔지니어링이다.

**Notion API rate limit**
- Notion API는 초당 3회 요청 제한이 있다.
- 폴링 방식은 rate limit 초과 위험이 있고, 불필요한 API 호출이 발생한다.

**명확한 상태 관리**
- 수동 Import는 "언제 데이터가 가져와졌는지"가 명확하다.
- 실시간 동기화는 캐시 상태와 실제 Notion 상태의 불일치를 감지하기 어렵다.

**Vercel 환경 제약**
- Vercel serverless 환경에서 cron job은 유료 플랜이 필요하다.
- 수동 Import는 추가 인프라 없이 동작한다.

### 운영 방식
- Notion에서 콘텐츠를 수정 → `/admin/import`에서 Import 버튼 클릭 → 즉시 반영
- 이 흐름은 1인 사용자에게 충분히 실용적이다.

## 4. 개인 프로젝트에서 이 구조가 적합한 이유

### 전체 구조 요약
```
Notion (편집) → [수동 Import] → JSON 캐시 → SSG 렌더링 → 공개 페이지
```

### 왜 개인 프로젝트에 맞는가

**외부 의존성 최소화**
- 별도 데이터베이스 없이 JSON 파일 캐시로 동작
- Vercel 무료 플랜으로 배포 가능
- 운영 비용 제로

**유지보수 부담 없음**
- 동기화 로직이 단순함 (fetch → write → done)
- 복잡한 상태 관리, 충돌 해결 로직 없음
- 문제 발생 시 캐시 파일을 삭제하고 다시 Import하면 됨

**Notion이 CMS 역할**
- Notion의 강력한 편집기와 데이터베이스 기능을 그대로 활용
- 별도 관리자 UI 개발 불필요
- 콘텐츠 작성/편집은 Notion에서, 공개 렌더링은 이 서비스에서 담당

**점진적 확장 가능**
- JSON 캐시 → SQLite 전환: `lib/cache/` 내부만 변경
- 수동 Import → 자동화: cron job 추가만으로 가능
- 1인 → 다중 사용자: OAuth 구조이므로 확장이 자연스러움

## 5. JSON 파일 캐시를 선택한 이유

### 결정
SQLite 대신 **JSON 파일 캐시**로 시작한다.

### 이유
- 설정 복잡도 없음 (드라이버 설치, 스키마 정의 불필요)
- 수백 개 수준의 포스트/노트에서 성능 문제 없음
- 캐시 내용을 텍스트 파일로 직접 확인 가능 (디버깅 용이)
- 문제 발생 시 파일 삭제 후 재Import로 초기화 가능

### 한계 및 전환 조건
- 항목이 1,000개를 초과하거나
- 복잡한 필터/정렬 쿼리가 필요해지면
- SQLite(`better-sqlite3`)로 전환. `lib/cache/` 내부만 교체.

## 6. 직접 블록 렌더러를 구현하는 이유

### 결정
`notion-to-md` 등 기존 라이브러리 대신 `NotionBlock.tsx` 컴포넌트를 직접 구현한다.

### 이유
- `notion-to-md`는 Markdown 문자열을 반환 → React 컴포넌트 수준 커스터마이징 어려움
- TOC 생성(heading에 id 부여), shiki 코드 강조, 코드 복사 버튼은 직접 구현 시 자연스럽게 통합
- 지원 블록 타입을 명시적으로 제어 → 미지원 타입 fallback 처리 명확
- 외부 라이브러리 버전 변경에 따른 영향 없음
